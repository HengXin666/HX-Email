# HX Email 重写设计说明

## 背景

本项目将以 `ref/outlookEmailPlus` 作为功能和行为参考进行重写。旧项目是一款面向注册与验证场景的邮箱管理器，不是通用邮件客户端；它的核心价值在于管理邮箱资源、提取验证码、提供邮箱池和临时邮箱能力。

重写后的后端使用 Python FastAPI 与 uv，默认 SQLite。前端使用 React、TypeScript、shadcn/ui 与 Tailwind，黑夜主题，整体工作台风格参考 `ref/web`，登录/注册页视觉参考 `ref/HX-ANiMe/HX-ANiMe-Web/src/views/LoginView.vue`，但必须重写，不复用旧前端代码。

## 设计原则

- 旧项目已有能力作为行为参考，除明确调整点外尽量沿用。
- 新增能力围绕“可用邮箱”扩展，不把别名邮箱和平台绑定硬塞进旧账号字段。
- 后端遵循深模块原则：FastAPI 路由保持薄层，复杂规则收敛在领域模块内部。
- 前端页面保持薄层，通过清晰 API 使用领域能力，不复制业务规则。
- 第一阶段优先做核心管理闭环，开放 API、浏览器扩展和插件化能力后置。

## 核心领域模型

### 用户

系统支持本地多用户。用户之间数据隔离，不引入团队和共享资源。同一个真实邮箱地址可以被不同用户各自添加和管理，形成彼此独立的数据副本。

管理员由环境变量提供初始用户名和密码。注册默认关闭，管理员可在后台设置中开启或关闭注册。普通用户只能管理自己的业务数据；管理员第一阶段只管理用户和系统开关，不接管普通用户业务数据。

### 邮箱账号

邮箱账号是收信能力和凭据配置的容器，例如 Outlook OAuth、普通 IMAP 或临时邮箱服务记录。邮箱账号至少拥有一个主邮箱地址，也可以关联多个真实别名邮箱地址。

邮箱账号不是平台绑定和验证码读取的业务单位。验证码读取、平台绑定、邮箱池领取和前端筛选都以可用邮箱为单位。

### 可用邮箱

可用邮箱是用户可以拿去注册平台、接收验证码、加入邮箱池或绑定用途的具体邮箱地址。它包括：

- 主邮箱地址：邮箱账号默认拥有的原始邮箱地址。
- 别名邮箱地址：已在邮箱服务商后台配置、由某个邮箱账号代收邮件的完整邮箱地址。
- 临时邮箱地址：由 CF 临时邮箱 Provider 创建或导入的地址，没有所属邮箱账号。

可用邮箱地址在同一用户的数据空间内唯一，不同用户可以各自管理相同邮箱地址。

别名邮箱地址不是 `user+tag@example.com` 这类 Plus 子地址。旧项目的 Plus 子地址兼容读取能力可以保留，但它不等同于新系统的真实别名邮箱地址。

### 验证码读取

读取某个可用邮箱的验证码时，必须按该可用邮箱的收件地址过滤。读取别名邮箱时，不能静默返回同一邮箱账号下其他地址收到的验证码。如果上游无法提供可靠收件地址，只能返回可疑结果或提示用户确认，不能伪装成确定命中。

### 平台与平台绑定

平台是用户用可用邮箱注册或登录的外部网站、应用或服务。平台名称在同一用户数据空间内唯一且大小写敏感。

平台绑定记录某个可用邮箱被用于哪个平台。一个可用邮箱可以绑定多个平台，一个平台也可以被多个可用邮箱绑定，但同一用户下同一组可用邮箱与平台只能有一条绑定记录。

平台绑定支持备注和状态。状态包括：

- `active`：正常使用中
- `pending_verification`：待验证
- `risk`：有风险
- `disabled`：不可用
- `archived`：已归档

系统可以从邮件发件人、主题或正文推测平台候选，但候选只用于提示用户，不能自动创建平台或平台绑定。用户确认后才写入绑定关系。

## 沿用旧项目能力

以下能力以 `ref/outlookEmailPlus` 为功能参考，第一阶段根据核心优先级取舍实现深度：

- 邮箱账号管理：Outlook OAuth、普通 IMAP、自定义 IMAP。
- 分组、标签、搜索、分页、导入导出。
- 邮件读取、邮件详情、验证码和验证链接提取。

- CF 临时邮箱：创建、读取、删除、接入邮箱池。
- 设置、登录、CSRF/基础安全、错误结构和审计思路。

第一阶段后置：

- 开放 API 的完整鉴权、Key 管理、IP 白名单、速率限制。
- 浏览器扩展。
- 临时邮箱 Provider 插件化，核心版只支持 CF。
- Telegram/Webhook/Email 通知。
- 一键更新与部署探测。
- AI 验证码增强。
- OAuth Token 工具。

## 后端模块边界

推荐模块：

- `auth`：本地用户、管理员初始化、注册开关、会话/JWT。
- `accounts`：邮箱账号凭据、主邮箱创建、分组和标签沿用行为。
- `usable_emails`：主邮箱、别名邮箱、临时邮箱的统一业务入口。
- `platforms`：平台、平台绑定、绑定状态、邮件平台候选。
- `verification`：验证码读取、收件地址过滤、提取策略。
- `mail_pool`：以可用邮箱为单位的池领取与状态机。
- `temp_mail`：CF 临时邮箱 Provider。
- `settings`：用户级和系统级设置。

FastAPI 路由只负责鉴权、HTTP 入参/出参和错误映射。跨模块操作通过深模块接口完成，不让路由拼接业务规则。

## 前端结构

第一阶段页面：

- 登录/注册页：参考 HX-ANiMe 的双态面板和背景视觉，用 React/Tailwind 重写。
- 可用邮箱工作台：主列表支持类型、状态、平台、分组、标签、关键词筛选。
- 邮箱账号详情：凭据、主邮箱、别名邮箱、分组、标签、状态。
- 平台管理：平台列表、创建、编辑、搜索。
- 平台绑定：在可用邮箱详情中绑定平台、设置状态和备注。
- 验证码读取：从可用邮箱触发读取，展示命中邮件、验证码、链接和收件地址判断。
- CF 临时邮箱：创建、查看、读取验证码，并进入可用邮箱列表。
- 邮箱池：按可用邮箱领取、释放、完成和查看状态。
- 管理后台：用户管理、注册开关、管理员修改用户名/密码。

黑夜主题为唯一主题。工作台应以密集、可扫描、面向重复操作的布局为主，不做营销式首页。

## 验收原则

- 每个核心能力都要有后端集成测试和前端可操作路径。
- 多用户隔离必须覆盖到邮箱账号、可用邮箱、平台、平台绑定、邮箱池和验证码历史。
- 别名邮箱必须按真实完整地址处理，不能被 Plus 子地址逻辑替代。
- 平台绑定必须支持状态和备注。
- 旧项目行为参考要以用户可见能力为准，不复制旧 Flask 分层和旧前端实现。
