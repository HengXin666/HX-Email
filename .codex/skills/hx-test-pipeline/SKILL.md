# HX Test Pipeline

从业务风险推导测试，不让用户选择其不了解的框架、层级或覆盖率目标。

## 自动探索

1. 读取仓库约束、产品文档、入口、路由、数据模型、状态转换、权限和外部调用。
2. 还原核心业务路径，并按影响与发生概率识别：数据损坏、越权、重复副作用、并发、失败恢复、协议漂移和用户可见错误。
3. 扫描现有测试、脚本、包管理器和 CI。能复用就不新增配置，能补现有测试文件就不新建测试框架。
4. 新项目建立严格而最小的测试骨架；老项目先保护当前行为和本次变更，逐步扩大，不要求一次偿还历史债。

## 分层设计

- static：格式、类型、Schema、模板渲染和依赖边界，是所有层之前的基础门禁。
- S1：纯业务规则、状态转换、组件状态和失败分支；进程内运行，禁止网络。
- S2：API、数据库、adapter、工具契约和稳定小型 eval；使用临时资源、stub 和固定 fixture。
- S3：完整应用或安装流程，覆盖成功主路径、权限拒绝和可恢复错误；外部服务仍可控。
- S4：真实模型、OAuth、MCP、供应商和网络漂移；专用账号、成本上限、定时运行，不阻塞普通 PR。

Agent Skill 仓库同时验证 description 的正/负触发和触发后的输出/副作用；同一案例用干净上下文比较 with-skill 与 baseline。真实失败优先固化到 S1-S3。

## 最少文件原则

- 优先使用项目已有的 `test`、`verify`、`check`、pytest、Vitest/Jest 或 Playwright 入口。
- 缺少统一入口时最多新增一个适配项目的 `scripts/verify.sh`；CI 直接调用它。
- 测试放入已有测试目录，fixture 靠近对应测试；不为展示分层而创建空目录、重复配置、测试说明文档或独立清单。
- 只有真实外部 canary 需要单独凭证和调度时才增加独立入口。

## 执行

先运行现有测试取得基线，再实现最小测试和修复。使用本 Skill 自带的零依赖验证器做最终兜底：

```bash
python skills/hx-test-pipeline/scripts/verify.py --root . --strict
```

安装在 `.codex/skills` 时使用对应路径。验证器会检查 Skill/脚本结构并自动复用仓库现有验证入口；它不替代为业务新增的断言。

失败时自行定位、修复本次问题并重跑。最终只汇报实际命令、各层结果、未验证的真实外部边界和剩余风险。

## 完成条件

- 每个高风险业务行为都有可观察断言，而不是只断言 mock 被调用。
- 本地和 CI 使用同一验证入口，SKIP 不伪装成 PASS。
- 老项目不因接入测试体系产生大规模无关重构。
- S4 与 PR 门禁分离，并能把真实失败回灌成确定性回归。
