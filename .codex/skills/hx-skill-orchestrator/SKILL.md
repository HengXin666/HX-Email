# HX Skill Orchestrator

把用户目标转成完整交付。不要让用户替你选择其不了解的工具、测试层级或严格度。

## 一键流程

1. 读取仓库内 `AGENTS.md` 等约束，扫描 Git 状态、源码、文档、依赖、锁文件、CI、测试和外部服务。
2. 自动判断项目状态：缺少有效业务代码时按新项目初始化；已有代码或工具链时按老项目渐进接入。不要要求用户选择模式。
3. 探索业务：从入口、路由、数据模型、状态转换、权限、外部调用和错误处理还原核心用户路径与风险。
4. 在相邻 Skill 目录中找到并读取需要的 `SKILL.md`，按以下顺序调度：
   - 技术栈缺失或需要新增依赖时使用 `hx-libs-sentaku`；已有项目优先沿用现有栈。
   - 使用 `hx-init` 安装或补齐最小工程约束。
   - 使用 `hx-test-pipeline` 设计并落地业务专属测试体系，最后执行统一验证。
5. 直接完成安全、项目内、可回滚的修改。保留用户已有文件和未提交改动，只做审阅式最小 patch。
6. 验证失败时自行定位并修复本次引入的问题，重跑受影响层和最终门禁；不要把常规修复选项重新抛给用户。
7. 最后只汇报做了什么、验证结果、无法自动完成的外部事项和真实剩余风险。

## 自动决策

- 从锁文件选择包管理器，从已有脚本选择 lint、类型检查和测试框架。
- 老项目默认渐进接入：复用现有工具，针对新增和变更建立门禁，不要求一次偿还全部历史债。
- 新项目默认严格初始化：建立最小目录、测试入口、CI 和业务主路径样例。
- 不为了“标准化”替换能工作的框架、格式化器或测试库。
- 没有明确需求时不增加 ORM、状态库、动效库、节点图库或真实外部测试依赖。
- 测试数量由风险决定，不机械追求覆盖率；优先验证业务不变量、权限、失败恢复和外部契约。
- 调度器不递归调用自身，也不声称未实际执行的 Skill 或测试已经完成。

## 仅在这些情况询问

- 需要用户提供账号、token、测试环境或付费额度。
- 将执行生产写入、不可逆迁移、删除数据、发布或其他外部副作用。
- 仓库事实支持多个互斥业务方向，选择会显著改变产品行为。
- 用户已有改动与必要修改直接冲突且无法安全合并。

除此之外自行探索、采用最保守且可回滚的方案继续完成。

## 完成条件

- 项目能够通过一个统一验证命令完成静态检查和确定性测试。
- 测试体系覆盖该项目真实业务主路径、拒绝路径、恢复路径和关键外部边界。
- 新老项目均保留一致的本地与 CI 入口，真实外部 canary 与 PR 门禁分离。
- 最终输出不要求用户补做本可由 AI 自动判断和执行的步骤。
